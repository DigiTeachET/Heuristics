The ones everyone mixes up

These three heuristics are close enough to cause arguments. Here is how to tell them apart.

3. User control and freedom

The mistake has already happened. The user clicked the wrong thing. Your job now is to give them a way out — an undo, a cancel, an emergency exit. This heuristic is about escape after a mistake.

Classic example: no undo button after deleting a file. The action happened — where is the exit?

5. Error prevention

The mistake has not happened yet. This heuristic is about stopping it before it occurs — through careful design, confirmation steps, or constraints that make the wrong action impossible or at least awkward. This heuristic is about stopping mistakes before they happen.

Classic example: an 'Are you sure you want to delete this?' dialogue before the action completes. The mistake is intercepted mid-flight.

9. Help users recognise, diagnose, and recover from errors

The mistake has happened and an error message has appeared. This heuristic is about whether that message is actually useful. Does it explain what went wrong in plain language? Does it tell the user how to fix it? Or does it just say 'ERROR_CODE_7741' and leave them stranded?

Classic example: 'Incorrect password. Please try again or reset your password here.' — clear, human, and actionable.

The quick test

Ask yourself: when in the sequence of events does the heuristic apply?

Before the mistake — Error Prevention (heuristic 5)
During the mistake — User Control and Freedom (heuristic 3)
After the mistake, reading an error message — Error Recovery (heuristic 9)
